# nPos / POS — จัดลำดับขายดี + โหมดเรียงเมนู

อัปเดต: วางแผน · ยังไม่ชิปโค้ด  
แนวคิด: **เก็บสถิติขายดีไปเรื่อยๆ** · **จัดลำดับที่หลังร้าน** · **หน้า POS แสดงตามโหมด** (fix หรือ กลุ่มขายดี)

## ความเข้าใจที่ล็อก

| หัวข้อ | ความหมาย |
|--------|----------|
| **เก็บข้อมูล** | นับ qty ขายต่อ `menuItemId` (+ หมวด) ไปเรื่อยๆ — ไม่ใช่แค่รายงานวันเดียว |
| **หน้าต่าง** | ช่วงแรก **7 วัน** · พอมีข้อมูลนิ่งขยายเป็น **14 วัน** ได้ (ค่าคอนฟิกหลังร้าน) |
| **ที่คำนวณ** | **หลังร้าน (BO / Cloud)** รวมแล้วเขียน **ตารางจัดลำดับ** |
| **Local-first** | เครื่องนับตอนขายสำเร็จทันที → ซิงก์ขึ้นหลังร้าน → ดึงตาราง rank มากับเมนู/แคช |
| **จังหวะเลื่อน** | **ไม่เลื่อนทุกบิลกลางจอขาย** — ใช้ตอนเปิดกะ / รีเฟรชเมนู / ซิงก์ snapshot |
| **โหมดเรียง (หลังร้านตั้ง 2 แบบ)** | **`fix`** = ลำดับคงที่/มือ · **`bestsellers`** = กลุ่ม+เมนูขายดีจริงอัตโนมัติ |
| **หน้า POS** | โหมด `bestsellers` แสดง **ขายดีของจริงเท่านั้น** (จากตาราง rank) — ไม่ปั้นจากธง `recommended` |
| **อย่าทับ `sortOrder` ถาวร** | แยกชั้นแสดง (`salesRank` / display order) · มือลากในโหมด fix ยังใช้ได้ |

## โหมดเรียง 2 รูป (หลังร้านเลือก 1)

### 1) `fix` — ลำดับคงที่ / มือ
- ใช้ `sortOrder` + (ช่วงนี้) fixed category names ถ้ายังเปิดอยู่
- ลากจัดที่ BO / หมวดบน POS ได้ตามเดิม
- **ไม่** ใช้สถิติขายดีจัดหน้า

### 2) `bestsellers` — กลุ่มขายดี
1. รู้ **หมวดขายดี** และ **เมนูขายดี** จากตาราง rank (หน้าต่าง 7→14 วัน)
2. **หมวดขายดีขึ้นหน้า** (เรียงตามคะแนนหมวด)
3. **ในแต่ละหมวด** เมนูขายดีขึ้นบน (เรียงตามคะแนนเมนู)
4. เสมอกัน → `sortOrder` มือ → ชื่อ
5. หมวด/เมนูที่ยังไม่มีสถิติพอ → ต่อท้ายในกลุ่ม “ยังไม่พอข้อมูล” (ไม่กระโดดสุ่ม)

หน้า POS (native + เว็บขาย) อ่านโหมดจากหลังร้านแล้วเรียงอัตโนมัติ — พนักงานไม่ต้องลากเองในโหมดนี้

## ตารางจัดลำดับ (เป้าเก็บที่ Firebase)

ตัวอย่างโครง (ชื่อจริงตอนลงมืออาจปรับ):

```
meta/posMenuRank  หรือ  posItemStats/{menuItemId}
{
  windowDays: 7 | 14,
  computedAt: <ms>,
  modeDefault: "fix" | "bestsellers",  // หรือเก็บใน shop settings
  categories: [{ categoryId, score, rank }],
  items: [{ menuItemId, categoryId, qty, score, rank }]
}
```

ตั้งโหมดที่ร้าน: `posSettings.menuArrangeMode = "fix" | "bestsellers"`

## เฟสงาน

### R0 เอกสาร + สัญญาโหมด
- [x] ล็อก 2 โหมด `fix` / `bestsellers`
- [x] ล็อกหน้าต่าง 7 วัน → ขยาย 14 ได้
- [x] ล็อก: POS โหมดขายดี = ข้อมูลจริงเท่านั้น · ไม่เลื่อนกลางบิล
- [x] ลิงก์จาก [npos-remaining-checklist.md](./npos-remaining-checklist.md)

### R1 Local นับขายดี
- [ ] ตอนขายสำเร็จ (native + เว็บ) +qty ตาม `menuItemId` / `categoryId`
- [ ] void / ทำลายบิล −qty ให้ตรง
- [ ] เก็บ local-first (prefs / ไฟล์เล็ก) อ่านได้ตอนออฟไลน์

### R2 ซิงก์ + รวมที่หลังร้าน
- [ ] ส่ง delta หรือ derive จาก `posSales` ในหน้าต่าง N วัน
- [ ] คำนวณคะแนนเมนู + คะแนนหมวด (qty หลัก)
- [ ] เขียนตารางจัดลำดับ + `computedAt`
- [ ] คอนฟิก `windowDays` 7 แล้วขยาย 14 ได้โดยไม่รีเซ็ตโค้ด

### R3 ตั้งโหมดที่หลังร้าน
- [ ] UI หลังร้าน: สลับ **แบบ fix** / **แบบกลุ่มขายดี**
- [ ] โชว์พรีวิวลำดับจากตาราง rank (อ่านอย่างเดียวในโหมด bestsellers)
- [ ] ในโหมด fix คงลากมือ / fixed list ตามนโยบายเดิม

### R4 หน้า POS เรียงตามโหมด
- [ ] `nposMenuSnapshot` (+ แคชเว็บ) ส่ง `menuArrangeMode` + rank
- [ ] native Sell + เว็บขาย: ถ้า `bestsellers` → หมวดขายดีหน้า · เมนูขายดีบน
- [ ] ถ้า `fix` → พฤติกรรมเดิม (sortOrder / fixed)
- [ ] ไม่รีเรียงกลางตะกร้าเปิดอยู่ · ใช้ตอนเปิดกะ / รีเฟรชเมนู
- [ ] โหมด bestsellers **ไม่** ใช้ธง `recommended` แทนสถิติ

### R5 กันพัง + นำร่อง
- [ ] ข้อมูลน้อยกว่าเกณฑ์ → ไม่บังคับดันสุ่ม · ผสม fix ชั่วคราวหรือต่อท้าย
- [ ] ปักหมุดมือ (optional) ทับ rank ได้ทีหลัง
- [ ] เทส: ออฟไลน์ยังขายได้ · ออนไลน์แล้ว rank ตามหลังร้าน
- [ ] คนเทสเคาน์เตอร์ยืนยันว่า “ขายดีบนจอ” ตรงยอดจริง

## นอกสcope รอบนี้
- เซ็นชื่อสลิป / KDS / ทานที่ร้าน-รับกลับ
- เปลี่ยน `sortOrder` ใน Firestore อัตโนมัติทุกคืน (ใช้ชั้น rank แทน)
- AI เดาเมนูแนะนำโดยไม่มีบิลจริง
