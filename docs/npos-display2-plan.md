# nPos — จอสอง (จอลูกค้า) · ตรวจสเปก + รายงานหลังร้าน

สถานะ: **แผนออกแบบ** (พักลงโค้ดหนัก — รอบถัดไปค่อยลงมือ)  
หลัก: พนักงานไม่ต้องรู้ศัพท์เทคนิค · **สเปก/สถานะจอส่งผ่านระบบรายงาน** (`nposDiagnose` + `nposOpsLog`)

---

## 1) คำตอบสั้น — พฤติกรรมจอสองตอนนี้

### POS เดิม (เว็บ / Chrome บนแท็บเล็ต)
- จอสองมักอาศัย **หน้าต่าง/โหมดหลายจอของระบบ** หรือแอปเดิมที่ถือ `Presentation`
- **ไม่การันตี** ว่าพับแอปแล้วยังค้างบนจอสอง — ส่วนใหญ่พับ/สลับแอปแล้วจอสองว่างหรือกลับโชว์ launcher
- ถ้าพนักงาน “เปิด POS ค้างไว้ตลอด” = แอปยังอยู่โฟร์กราวด์/ไม่ถูกฆ่า → จอสองถึงจะโชว์ต่อเนื่อง

### nPos ใหม่ (native · `Presentation`)
- ใช้ `CustomerAmountPresentation` / `NumberPresentation` ส่งไปจอที่ไม่ใช่จอหลัก
- ผูกกับ Activity ที่เปิดอยู่ (เช่นหน้าขาย)
- **พับแอป / ออกจากหน้าขาย / destroy Activity → จอสองถูกปิด** (โค้ด `dismissCustomer()` ใน `onDestroy`)
- เปิด nPos ขึ้นมาใหม่ → **แสดงกราฟิกบนจอสองได้** ถ้าระบบเห็นจอรอง (HDMI / จอลูกค้าในตัว)

### ข้อจำกัดสำคัญตอนร้านยังรัน POS เดิมคู่ขนาน
- จอสองมักเป็น “เจ้าของคนเดียว” — ถ้า POS เดิมยังถือจอสองอยู่ nPos อาจโชว์ไม่ได้หรือแย่งกัน
- ช่วงนำร่อง: ตกลงว่า **เครื่องไหนใช้แอปไหนเป็นเจ้าของจอสอง** (อย่าเปิดสองแอปแข่งกัน)

---

## 2) เป้าหมายระบบ (เมื่อกลับมาพัฒนา)

```
เปิดแอป / กลับโฟกัส / เสียบจอใหม่
        │
        ▼
  DisplayProbe ขยาย  →  สเปกจอทุกจอ (โดยเฉพาะจอ 2)
        │
        ├─ บันทึก snapshot → nposDiagnose (หลังร้าน “ตรวจเครื่อง”)
        ├─ สถานะสั้น / ผิดปกติ → nposOpsLog (ไทม์ไลน์)
        └─ (ถ้าขายอยู่) วาด UI จอลูกค้าตาม orientation จริงของเครื่องนั้น
```

พนักงานเห็นแค่: “จอลูกค้าพร้อม / ไม่พร้อม”  
เจ้าของ/agent เห็นสเปกครบในหลังร้าน

---

## 3) สิ่งที่ต้องวัดจากจอสอง (และจอหลัก)

ขยายจาก `DisplayProbe` ที่ตอนนี้มีแค่ `number · displayId · primary · name`

| ฟิลด์ | ทำไมต้องมี |
|-------|------------|
| `widthPx` / `heightPx` | ออกแบบ layout |
| `densityDpi` / `scaledDensity` | ตัวอักษรใหญ่พอไหม |
| `refreshHz` (ถ้าได้) | วินิจฉัยกระตุก |
| `orientation` = `portrait` \| `landscape` | **เคสร้านแนวนอน vs ดีไซน์แนวตั้ง** |
| `rotation` (0/90/180/270) | หมุนจริงของแผง |
| `isPrimary` | แยกจอขาย vs จอลูกค้า |
| `name` / `displayId` | อ้างอิงซ้ำ |
| `presentationOk` | ลอง `Presentation.show()` สำเร็จไหม (เทสเงียบ) |
| `lastProbeAt` | ความสดของข้อมูล |

สรุปหลังร้านที่อ่านง่าย:

| สถานะ | ความหมาย |
|--------|----------|
| `ok` | มีจอสอง + โชว์ Presentation ได้ |
| `missing` | มีแค่จอเดียว |
| `orientation_mismatch` | แอปคาดแนวตั้ง แต่เครื่องเป็นแนวนอน (หรือกลับกัน) |
| `show_failed` | มีจอแต่โชว์ไม่ได้ (ถูกแอปอื่นครอง / สิทธิ์ / driver) |

---

## 4) ช่องทางรายงาน (ย้ำตามที่ขอ)

| ช่อง | ใช้กับ | ตัวอย่าง |
|------|--------|---------|
| **`nposDiagnose`** | snapshot สเปกเต็ม (หลัก) | รายการจอ + ขนาด + orientation + ผลเทสโชว์ |
| **`nposOpsLog`** | เหตุการณ์สั้น / ผิดปกติ | `display` · “จอลูกค้าแนวนอน — layout แนวตั้งอาจเพี้ยน” |
| **`posDevices`** (เบา) | สถานะเครื่องออน | เช่น `customerDisplay: ok\|missing\|warn` หนึ่งฟิลด์ |

**ไม่**พึ่งให้พนักงานถ่ายรูปหรือพิมพ์สเปกมา  
**ไม่**โชว์ตัวเลขเทคนิคบนจอขาย

---

## 5) เคส orientation ที่คุณยก

```
ดีไซน์จอลูกค้า = แนวตั้ง (portrait)
เครื่องหน้าร้านจอสอง = แนวนอน (landscape)
        │
        ▼
probe ส่ง orientation_mismatch
        │
        ├─ หลังร้าน: เห็นชัดว่าเครื่องนี้ layout ต้องเป็นแนวนอน
        └─ แอป (รอบถัดไป): เลือก layout ตาม orientation จริง — ไม่บังคับแนวตั้งทุกเครื่อง
```

กฎที่แนะนำ:

1. **รายงานก่อน** — รู้สเปกจริงทุกเครื่อง  
2. **layout ตามเครื่อง** — มี `presentation_customer_portrait` / `_landscape`  
3. ไม่สมมติว่าทุกสาขาเหมือนกัน

## 6) เฟสลงมือเมื่อกลับมาทำ (เบา → หนัก)

### D1 — Probe + รายงาน (ไม่แตะ UI ขาย)
- ขยาย `DisplayProbe` ให้เก็บตารางด้านบน
- ส่งใน diagnose อัตโนมัติ (เปิดแอป / กลับโฟกัส / เสียบจอ)
- ops เมื่อ `missing` / `show_failed` / `orientation_mismatch`
- หลังร้าน: การ์ดจอใน “ตรวจเครื่อง” โชว์ขนาด + แนว + สถานะ

### D2 — สถานะสั้นบนเครื่อง + ใน `posDevices`
- ฟิลด์ `customerDisplay` สั้นๆ
- พนักงานเห็น “จอลูกค้า: พร้อม / ไม่พบ”

### D3 — Layout ตาม orientation
- เลือก layout ตามสเปกที่ probe ได้
- ยังส่งผล “แสดงผลถูกต้องไหม” กลับ ops หลังเทสเงียบ

### D4 — ช่วงคู่ขนานกับ POS เดิม
- คู่มือสั้น: ปิด/พับ POS เว็บก่อนเปิด nPos บนเครื่องที่ใช้จอสองร่วม
- (ถ้าต้องการ) ตรวจว่า Presentation ล้มเพราะจอถูกครอง → ops ชัดเจน

---

## 7) สิ่งที่มีอยู่แล้ว (ไม่เริ่มจากศูนย์)

| มีแล้ว | ช่องว่าง |
|--------|----------|
| `DisplayProbe.listDisplays` | ยังไม่มีขนาด / orientation / dpi |
| `CustomerAmountPresentation` บนหน้าขาย | dismiss เมื่อออกหน้า · ยังไม่รายงานสเปก |
| diagnose → หลังร้าน | payload จอยังบาง |
| ops `display` เมื่ออัปเดตจอลูกค้าล้ม | ยังไม่มี orientation_mismatch |

---

## 8) นอกขอบเขตรอบนี้

- ไม่ redesign กราฟิกจอลูกค้าเต็มรูปแบบจนกว่า D1 ได้ข้อมูลจริงจากเครื่องร้าน
- ไม่บังคับตัดเว็บขาย (N7) เพราะจอสอง
- ไม่สร้างช่องทางรายงานใหม่แยกจาก diagnose/ops

---

## 9) เกณฑ์ผ่านเมื่อทำ D1

1. หลังร้านเห็นจอ 2: กว้าง×สูง · orientation · ชื่อจอ  
2. เครื่องไม่มีจอสอง → สถานะ `missing` ในรายงาน (ไม่เงียบ)  
3. เครื่องแนวนอน vs ดีไซน์แนวตั้ง → ธง `orientation_mismatch`  
4. พนักงานไม่ต้องกดปุ่มเทคนิคเพื่อส่งสเปก
