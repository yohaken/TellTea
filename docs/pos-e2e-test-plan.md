# แผนงานทดสอบ POS แบบจำลองผู้ใช้ (E2E)

เป้าหมาย: ไม่ใช่แค่ pass/fail — **สังเกตความเหมาะสม** (ช้าไปไหม, คลิกแล้วเงียบ, error บนหน้า) สำหรับงานหน้าร้านที่ต้อง **รวดเร็ว**

เครื่องมือ: **Playwright** (จำลอง iPhone 13 / แท็บเล็ต) · รันใน CI กับ `out-pos` หลัง build

---

## งบเวลา (Performance budgets)

| ขั้นตอน | เป้า (warn) | ขีดล้ม CI (fail) | เหตุผล |
|--------|-------------|------------------|--------|
| โหลด → พร้อมขาย | 12s | 45s | ครั้งแรกอาจช้า, ครั้งถัดไปต้องเร็วขึ้น |
| คลิกเฟือง → หน้าเมนู | 2.5s | 8s | พนักงานไม่ควรรอ |
| หน้าเมนู auth + แท็บ | 6s | 20s | ไม่ควร permission error |
| เปิดขายกะ | 5s | 15s | ก่อนเริ่มคิวลูกค้า |
| โหลดกริดเมนู | 8s | 25s | ต้องเห็นเมนูก่อนรับออเดอร์ |
| แตะเมนู → ตะกร้า | **0.8s** | 3s | **สำคัญสุด** — รู้สึกทันที |
| popup ตัวเลือก → ตะกร้า | 1.2s | 4s | คิวยาวต้องไว |
| เงินสด → ยืนยันขาย | 2s | 8s | local-first ไม่ควรค้าง |
| ขาย ↔ เมนู ไปกลับ | 5s | 15s | สลับจัดการเมนูบ่อย |

ค่าอยู่ใน `scripts/pos-e2e-harness.mjs` (`POS_E2E_BUDGETS`)

## สิ่งที่สังเกตนอกจากเวลา

| อาการ | ความหมาย |
|-------|----------|
| คลิกแล้ว URL ไม่เปลี่ยน | routing พัง (เคยเจอ Next Link) |
| `Missing or insufficient permissions` | auth ก่อน Firestore |
| กริดเมนูว่าง | seed / sync / หมวดปิดหมด |
| `เชื่อมต่อไม่สำเร็จ` | Firebase env / auth |
| React #418 | hydration ไม่ตรง — ไม่บล็อก deploy แต่ควรแก้ |
| ปุ่มเงินสด disabled | ตะกร้าไม่เพิ่ม (แตะเมนูพัง) |
| ช้ากว่า warn แต่ไม่ fail | regression เตือนล่วงหน้า |

## แผนงานตามเฟส

### เฟส 1 — ทำแล้ว / กำลังรันใน CI

| สคริปต์ | ครอบคลุม | คำสั่ง |
|---------|----------|--------|
| `test-pos-nav-e2e.mjs` | เฟือง → เมนู → กลับขาย | `npm run test:pos-nav-e2e` |
| `test-pos-menu-e2e.mjs` | กลุ่มตัวเลือก + บันทึก | `npm run test:pos-menu-e2e` |
| `test-pos-sell-e2e.mjs` | เปิดกะ → แตะเมนู → เงินสด | `npm run test:pos-sell-e2e` |
| `test-pos-e2e-all.mjs` | รันทั้งชุด + JSON report | `npm run test:pos-e2e-all` |

### เฟส 2 — ถัดไป (ลำดับความสำคัญ)

| งาน | ทดสอบอะไร | สังเกต |
|-----|-----------|--------|
| **options-e2e** | เมนูที่มีท็อปปิ้ง → เลือก → ราคารวม | popup ช้า, บังคับเลือกผิด |
| **soldout-e2e** | กดค้าง → ของหมด → หายจากกริด | กับแตะสั้นไม่สับสน |
| **category-e2e** | สลับหมวด → เมนูถูกต้อง | แท็บ lag |
| **promptpay-e2e** | QR ขึ้นใน 2s | ไม่ทดสอบสแกนจ่ายจริง |
| **refresh-e2e** | F5 หน้าขาย → เมนูจาก cache | โหลดช้าโดยไม่มีเน็ต |

### เฟส 3 — ความทนทาน

| งาน | ทดสอบอะไร |
|-----|-----------|
| **offline-e2e** | offline ขาย → online → sync |
| **stuck-bill-e2e** | บิลค้าง → panel → retry |
| **multi-tap-e2e** | แตะเมนู 5 ครั้งติด → qty ถูก |
| **production-smoke** | ทุก 30 นาทีบน live (แยก workflow) |

### เฟส 4 — ไม่ทำอัตโนมัติ

- เครื่องพิมพ์ใบเสร็จจริง
- ลูกค้าสแกน PromptPay
- ทุกรุ่นแท็บเล็ตในร้าน

## วิธีรัน

```bash
# หลัง build — ทดสอบกับ static export (เหมือน CI)
npx --yes serve out-pos -l 8765 &
POS_E2E_URL=http://127.0.0.1:8765/pos/ npm run test:pos-e2e-all

# ทดสอบ production (หลัง deploy)
npm run test:pos-e2e-all
```

รายงาน JSON: `/tmp/pos-e2e-report.json` (เมื่อรัน all)

## นิยามผ่าน / ไม่ผ่าน

- **PASS** — ฟังก์ชันถูก + เร็วกว่า warn
- **WARN** — ฟังก์ชันถูก แต่ช้ากว่าเป้า (ไม่ fail CI ยกเว้นกำหนด `POS_E2E_STRICT=1`)
- **FAIL** — ฟังก์ชันพัง หรือช้ากว่า fail budget
- **OBS** — ข้อสังเกต (เมนูว่าง, hydration ฯลฯ)
